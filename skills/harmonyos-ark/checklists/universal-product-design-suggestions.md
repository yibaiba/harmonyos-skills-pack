# 通用建议：功能丰富度 + 深色模式 + 多端适配

> 适用于 HarmonyOS 应用提审前的产品质量自检。
> 配合 checklists/pre-submission-2025.md 和 topics/ux-standards.md 使用。
> 📎 更完整的通用质量体系见 `skills/universal-product-quality/` skill（含风险分级、评审打分表、发布清单）。

## 1. 功能点丰富度

### 1.1 业务链路要求
1. 至少 3 条完整业务链路
   - 每条链路都包含入口、处理过程、结果反馈
   - 示例链路：浏览→筛选→收藏、创建→编辑→删除、登录→使用→退出
2. 每条链路都包含异常分支
   - 弱网、拒权、空数据、服务异常至少覆盖 2 种
   - 异常状态必须有明确的用户提示（Toast/空态页/重试按钮）
3. 每条链路都有可验证结果
   - 用户可看到明确成功或失败状态
   - 审核账号可复现全部链路

### 1.2 功能深度要求
- 每个 Tab 页必须有实际内容，禁止"敬请期待"占位
- 列表页必须支持下拉刷新 + 上拉加载更多（数据量 > 1 屏时）
- 搜索/筛选功能必须返回有意义的结果（禁止假数据或全为空）
- 表单提交必须有输入校验 + 提交反馈 + 错误提示
- 设置页必须有至少 1 个真实可切换的配置项

### 1.3 反面模式（审核常见拒绝理由）
- 只有入口页面，没有真实业务闭环
- 所有页面展示相同的 mock 数据
- 点击按钮无响应或弹出"暂未开放"
- 功能文案与实际行为不一致

## 2. 深色模式

### 2.1 基础要求
1. 支持系统主题跟随
   - 使用系统颜色 token（`$r('sys.color.xxx')`），禁止硬编码颜色
   - 不建议仅保留手动开关而忽略系统主题
2. 保证对比度
   - 文本/背景对比度 >= 4.5:1（WCAG AA 标准）
   - 图标、按钮在深色背景下可辨识
   - 禁止在深色模式下使用纯黑背景 + 纯白文字（太刺眼）
3. 图片与媒体适配
   - 图片资源准备 light/dark 双套，或使用 ColorFilter 调整
   - 阴影使用透明度而非固定灰度值
   - 地图、图表等第三方视图需单独验证深色效果

### 2.2 重点验证页面
- 首页、详情页、提交流程页、设置页
- 弹窗、Toast、对话框
- 加载态、空态、错误态页面
- 闪屏/启动页

## 3. 多端适配（手机/平板/PC）

### 3.1 断点布局
- 手机: < 600vp（单栏布局）
- 平板: 600vp - 840vp（双栏或侧边导航）
- PC: >= 840vp（多栏 + 效率操作）

### 3.2 布局策略
- 手机优先任务流，减少信息密度
- 平板增加并行信息（双栏详情、侧边导航）
- PC 增加效率操作（快捷键提示、批量操作、右键菜单）

### 3.3 交互一致
- 核心操作路径保持一致，不因设备变化而缺失功能
- 触控热区最小 48vp x 48vp
- 键鼠操作支持 hover 态和焦点高亮
- 导航方式适配：手机底部 Tab → 平板/PC 侧边栏

### 3.4 常见破版问题
- 平板横屏内容被拉伸变形
- PC 上大量留白，信息密度过低
- 文字截断或重叠
- 操作按钮超出屏幕可见区域

## 4. 提审前最小验收

### 4.1 功能验收
1. 三端都可完成核心业务链路
2. 每条链路包含正常流程 + 至少 1 个异常流程
3. 素材、文案、功能一致

### 4.2 体验验收
1. 深色模式下关键页面可读可操作
2. 弹窗/Toast 在深色模式下正常显示
3. 多端布局无截断、重叠、不可点击区域

### 4.3 审核验收
1. 审核账号可复现主要流程
2. 应用描述与实际功能一致
3. 截图/视频反映当前版本真实状态

## 5. 适用场景
- 新项目立项时定义产品基线
- 版本提审前做快速体检
- 团队评审时统一口径
- AI Agent 实现功能时的自检参考
