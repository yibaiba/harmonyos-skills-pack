# Universal Product Quality Skill - 一页式使用说明

## 目标

用最短时间建立“可发布”的统一质量门槛，避免功能单薄、深色模式缺陷、多端破版和提审阻塞。

## 适用范围

- 任意项目：App / Web / 桌面 / 小程序
- 任意技术栈：前端框架、原生、跨端方案均可

## 质量门流程图

```
  ┌─────────────┐
  │  新项目/新版本 │
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌────────────────────────┐
  │ 1. 判定风险等级 │────▶│ rules/risk-auto-        │
  └──────┬──────┘     │ classification.md       │
         ▼            └────────────────────────┘
  ┌─────────────┐     ┌────────────────────────┐
  │ 2. 执行检查清单│────▶│ checklists/pre-release  │
  └──────┬──────┘     │ + multi-platform-dark   │
         ▼            └────────────────────────┘
  ┌─────────────┐     ┌────────────────────────┐
  │ 3. 团队评审打分│────▶│ templates/team-review   │
  └──────┬──────┘     │ -scorecard.md           │
         ▼            └────────────────────────┘
     ≥85 且无 Blocker?
      ├── Yes ──▶ ✅ 发布
      └── No  ──▶ 🔁 修复 → 回到步骤 2
```

## 检查清单选择决策树

- **所有项目** → `checklists/pre-release-universal.md`（必选）
- 有深色模式或多端？→ 加选 `checklists/multi-platform-dark-mode.md`
- 涉及支付/账号/敏感数据？→ 风险等级 ≥ High，执行完整回归
- 纯展示/工具类？→ 风险等级 Low，执行基础检查即可

## 最小质量门 vs 完整质量门

| 维度 | 最小质量门（Low 风险） | 完整质量门（High 风险） |
|-----|---------------------|----------------------|
| 链路验证 | 3 条核心链路 happy path | 全部链路 + 异常分支 |
| 深色模式 | 首页 + 详情页检查 | 全页面 + 对比度测量 |
| 多端适配 | 主端通过即可 | 三端逐页检查 |
| 评审 | 1 人快速打分 | 团队正式评审会 |
| 耗时 | 30 分钟 | 2-4 小时 |

## 新项目 5 分钟上手

1. 复制 `templates/project-onboarding-template.md`
2. 填项目信息和 3 条核心链路
3. 跑两份检查清单并记录问题
4. 评审打分，输出发布结论

## 常见失败信号

- 只有页面展示，没有完整功能闭环
- 深色模式按钮/文本对比度不足
- 平板或桌面出现布局重叠、不可点击
- 文案、截图、实际功能不一致

## 集成建议

### PR Review 集成
- 在 PR 模板中添加质量检查勾选项（从 `pre-release-universal.md` E 节复制）
- 指定至少一位 reviewer 对照 scorecard 打分

### CI/CD 集成
- 将可自动化的检查（链接有效性、对比度、截图回归）加入 pipeline
- 将不可自动化的检查（业务链路、主观体验）安排为人工 gate

### Sprint 集成
- Definition of Done 中包含"质量清单通过"
- Sprint 末尾预留质量门时间（Low: 30min / High: 2-4h）

## FAQ

**Q1: 我的项目不需要深色模式，可以跳过吗？**
可以。只跑 `pre-release-universal.md`，评审中 B1 项标记为 N/A 并说明理由。

**Q2: 团队只有一个人，怎么做评审？**
用 scorecard 做自评，重点关注所有 🔴 Blocker 项。建议找一位外部同事做 15 分钟快速 cross-check。

**Q3: 已有 CI/CD 流水线，如何集成？**
将 `pre-release-universal.md` 的 E 节勾选表转为 CI 检查项（如截图对比、链路自动化测试），在发布前 pipeline 中作为 gate。

**Q4: Sprint 规划时如何使用？**
在 Sprint 最后一天预留 2-4 小时作为“质量门时间”。Sprint planning 时将质量检查作为 Definition of Done 的一部分。

**Q5: 分数在 70-84 之间，必须全部修复吗？**
只需修复所有 🔴 Blocker 项。🟡 Warning 项根据发布紧迫度取舍，但需记录为已知问题。

**Q6: 能用于 API / 后端项目吗？**
可以。将“页面”替换为“接口”，将“截图”替换为“请求/响应样例”，核心理念（链路闭环、异常覆盖、稳定性）同样适用。
