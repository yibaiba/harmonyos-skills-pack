# 功能丰富度主题（通用）

## Goal
确保产品不是"仅展示页面"，而是具备可用、可验证、可交付的功能闭环。

## Baseline
1. 至少 3 条完整业务链路
2. 每条链路包含成功与失败反馈
3. 每条链路至少覆盖 1 个异常分支
4. 审核账号或测试账号可复现全部链路

## 链路分类与要求

### CRUD 链路（创建/读取/更新/删除）
- 创建：输入校验 -> 提交 -> 成功/失败反馈
- 读取：加载态 -> 数据展示 / 空态 / 错误态
- 更新：编辑 -> 保存 -> 数据同步确认
- 删除：二次确认 -> 删除 -> 撤销（可选）
- 异常：网络中断、服务端错误、权限不足、并发冲突

### 浏览筛选链路
- 浏览 -> 搜索/筛选 -> 查看详情 -> 收藏/分享
- 异常：搜索无结果、筛选条件为空、分页加载失败

### 认证授权链路
- 登录 -> 使用核心能力 -> 退出/撤销
- 异常：密码错误、Token 过期、账号被禁、多设备互踢

### 设置偏好链路
- 查看设置 -> 修改配置 -> 即时生效确认
- 异常：设置项冲突、权限变更后的降级处理

## 功能完整度评分矩阵

| 等级 | 分值 | 标准 |
|------|------|------|
| 无 | 0 | 入口存在但点击无响应或显示"敬请期待" |
| 桩 | 1 | 有 UI 展示但使用 mock/硬编码数据 |
| 部分 | 2 | 正常流程可走通，但无异常处理 |
| 完整 | 3 | 正常+异常流程完整，状态反馈清晰 |

最低发布标准：核心链路（>=3 条）全部达到 3 分

## 边界情况覆盖清单

### 数据边界
- 空态：列表为空、搜索无结果、用户无内容
- 极端值：超长文本、超大数字、特殊字符
- 首次使用：无历史数据、无缓存、引导流程

### 环境边界
- 弱网/断网：请求超时、重试机制、离线缓存
- 权限拒绝：相机/存储/定位被拒后的降级提示
- 后台恢复：应用从后台切回时的状态保持

### 并发边界
- 重复提交：按钮防抖、请求去重
- 数据竞争：多页面共享数据时的一致性
- 中断恢复：操作中途退出后再进入的状态处理

## 功能深度 vs 广度决策指南

| 项目阶段 | 优先策略 | 说明 |
|---------|---------|------|
| MVP / 提审版 | 深度优先 | 3 条核心链路做到完整（3 分），其余暂不实现 |
| 正式版 | 平衡 | 核心链路完整 + 辅助功能至少部分（2 分） |
| 迭代版 | 广度扩展 | 逐步补齐辅助功能和边界情况 |

## Common Pitfalls
- 只有入口页面，没有真实业务闭环
- 只验证 happy path，没有异常流转
- 文案和实际功能不一致
- 所有页面展示相同的 mock 数据
- "敬请期待"占位页面过多
- 搜索/筛选返回空但无空态引导

## Practical Advice
- 每条链路至少准备 1 组测试数据
- 每条链路至少准备 1 条异常路径脚本
- 在发布前做一次全新安装验证（冷启动 + 无缓存）
- 用审核账号完整走一遍所有核心链路
- 截图或录屏留档，便于审核沟通和回归对比
