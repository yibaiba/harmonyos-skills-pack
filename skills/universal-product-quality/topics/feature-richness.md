# 功能丰富度主题（通用）

## Goal

确保产品不是“仅展示页面”，而是具备可用、可验证、可交付的功能闭环。

## Baseline

1. 至少 3 条完整业务链路
2. 每条链路包含成功与失败反馈
3. 每条链路至少覆盖 1 个异常分支
4. 审核账号或测试账号可复现全部链路

## 链路类型详解

### CRUD 链路
- 创建 → 查看 → 编辑 → 删除
- 异常：重复创建、编辑冲突、删除后撤销、字段校验失败
- 验证点：删除确认二次弹窗、编辑后数据持久化

### 搜索/筛选链路
- 输入关键词 → 展示结果 → 筛选/排序 → 查看详情
- 异常：无结果空态、特殊字符输入、超长关键词、筛选条件互斥
- 验证点：搜索防抖、结果高亮、筛选重置

### 设置/偏好链路
- 进入设置 → 修改选项 → 保存 → 验证生效
- 异常：未保存退出、恢复默认值、跨设备同步冲突
- 验证点：设置项即时生效 vs 需重启、回退能力

### 分享/导出链路
- 选择内容 → 分享/导出 → 接收方验证
- 异常：无网络分享、大文件导出超时、分享链接失效
- 验证点：分享预览、格式保真、权限控制

### 通知/消息链路
- 触发事件 → 收到通知 → 点击通知 → 到达目标页
- 异常：通知权限关闭、应用被杀后的离线通知、通知堆叠
- 验证点：通知内容准确、深链接跳转正确

## 每种链路的错误处理模式

| 链路类型 | 必备错误处理 |
|---------|------------|
| CRUD | 字段校验提示、并发冲突提示、删除二次确认 |
| 搜索/筛选 | 空结果态、请求超时、输入过滤/转义 |
| 设置/偏好 | 保存失败回滚、非法值拒绝、权限不足提示 |
| 分享/导出 | 网络中断重试、格式不支持提示、大小超限 |
| 通知/消息 | 权限引导、消息过期处理、重复通知去重 |

## 功能完成度评分矩阵

对每条链路，按以下标准评分：

| 分数 | 等级 | 标准 |
|-----|------|-----|
| 0 | 无 | 功能入口不存在或完全不可用 |
| 1 | 桩代码 | 有入口和 UI，但点击无响应或返回假数据 |
| 2 | 部分完成 | Happy path 可走通，但缺少错误处理和边界覆盖 |
| 3 | 完成 | 成功/失败/异常路径均已实现，可通过发布验收 |

> 发布门槛：所有 P0 链路 ≥ 3，P1 链路 ≥ 2，不允许存在分数为 0 的链路。

## 边界场景覆盖清单

- [ ] **空态**：首次使用、无数据、清空后的状态
- [ ] **首次运行**：引导流程、权限申请、初始化设置
- [ ] **权限拒绝**：相机/位置/通知等权限被拒后的降级体验
- [ ] **离线/弱网**：断网提示、缓存展示、重连恢复
- [ ] **并发操作**：多次快速点击、多端同时编辑、后台刷新
- [ ] **极端数据**：超长文本、特殊字符、极大/极小数值、空字符串
- [ ] **中断恢复**：操作中切后台、来电打断、内存回收后返回

## 功能深度 vs 广度取舍

```
  深度优先（推荐）          广度优先（风险高）
  ┌─────────────┐          ┌─────────────┐
  │ 3 条链路      │          │ 8 条链路      │
  │ 每条评分 3    │          │ 每条评分 1-2  │
  │ 错误处理完善  │          │ 只有 happy path│
  │ 边界覆盖充分  │          │ 多处桩代码    │
  └─────────────┘          └─────────────┘
  ✅ 可发布                 ❌ 不可发布
```

**原则**：宁可砍掉低优先级功能，也不要交付“处处能进、处处半成品”的产品。

## 反模式：“Demo 品质” vs “生产品质”

| 维度 | Demo 品质 | 生产品质 |
|-----|----------|---------|
| 数据 | 硬编码 / 假数据 | 真实接口 + 持久化 |
| 错误 | console.log 或静默吞掉 | 用户可感知的错误提示 + 重试 |
| 状态 | 只有成功态 | 加载 / 成功 / 失败 / 空态 |
| 边界 | 未处理 | 有防护且有降级 |
| 刷新 | 需手动刷新页面 | 自动刷新或实时同步 |

## Common Pitfalls

- 只有入口页面，没有真实业务闭环
- 只验证 happy path，没有异常流转
- 文案和实际功能不一致
- 功能入口存在但点击后跳转到空白页或 404
- 表单可提交但无后续流程（提交后无反馈、无列表更新）

## Practical Advice

- 每条链路至少准备 1 组测试数据
- 每条链路至少准备 1 条异常路径脚本
- 在发布前做一次全新安装验证
- 使用功能完成度矩阵量化评估，避免主观判断
- 优先完善 P0 链路深度，而非增加 P2 链路数量
